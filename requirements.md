# プロダクト企画書：ニーディア (Need+Idea)

**〜日々の「あったらいいな」を世界に放ち、誰かのコードになる〜**

---

## 1. プロダクトビジョン

> 「あなたの小さな不満は、誰かがまだ気づいていないプロダクトのタネだ」

ニーディアは、**エンジニアへの制作依頼サービスではありません。**

日常生活のなかでふと感じる「これ不便だな」「こんなの誰か作ってくれないかな」という気持ち——それ自体に価値があると考えています。誰もが専門家でなくていい。自分の言葉で、自分の目線で、不満やアイデアを世に放てばいい。

投稿されるのは「個人の発注」ではなく、**その人にしか気づけなかった視点**です。電車を毎日使う人だから気づく不便さ、子育て中の親だから感じる煩わしさ、一人暮らしを始めて初めて直面する面倒さ——そういった「生活の中からにじみ出るリアルな声」が集まる場所がニーディアです。

エンジニアはそこに刺激を受け、自分の作りたいものを見つける。投稿者は「自分の感覚が世界に届いた」体験を得る。ニーディアは、そのふたりをゆるやかにつなぐ**アイデアの広場**です。

既存の掲示板は「言いっぱなし」で終わり、GitHubは「技術の展示場」に留まっています。ニーディアは、アイデアが実装されるまでの「軌跡」を可視化し、投稿者には**「発想の価値」**を、開発者には**「社会実装の証明」**を与えることで、アイデアが循環し続ける世界を実現します。

---

## 2. 想定ユースケース

### A. アイデア投稿者（Need-er）

投稿者は「エンジニアに仕事を頼む人」ではありません。技術的な知識はゼロでもいい。投稿のハードルはツイートと変わらない。

- **シーン:** 毎朝の通勤中に電車が遅延して「もっと早く知れたらいいのに」とスマホを見る。
- **アクション:** ニーディアに「電車の遅延、もっとリアルタイムで教えてほしい」とそのまま投稿。
- **体験:** 同じことを感じていた100人が「欲しい！」を押す。開発者が拾い、アプリが生まれる。「あなたのひとことが100人に届きました」という通知とともに、自身の「発想スコア」が上がる。

**投稿のイメージ:**
- 「冷蔵庫にある食材から夕飯を提案してほしい」
- 「読んだ本の内容を忘れないようにまとめてくれるアプリがあれば」
- 「子どもの予定を家族全員でリアルタイム共有したい」
- 「会議中にメモを取りながら発言するのがつらい」

これらはすべて、その人が日常を生きているからこそ気づいた、その人にしか言えない言葉です。

### B. 開発者（Build-er）

- **シーン:** 新しい技術（Next.js等）を学びたいが、作るものが思いつかず、また「TODOアプリ」を作ろうとしている。
- **アクション:** ニーディアで「Need（需要）」が強く、かつ自分の技術スタックに合うアイデアを検索。
- **体験:** 投稿者と対話しつつ実装。リリース後、自分のプロフィールに「課題解決実績」が刻まれ、エンジニアとしての市場価値が可視化される。

## 3. UI/UXイメージ

### ① トップ画面（Marketplace of Ideas）

- **構成:** Twitterのようなタイムライン形式だが、各カードには「Needスコア（欲しいボタンの数）」と「想定技術スタック」がタグ付けされている。
- **UX:** 「注目度が高い未解決課題」がひと目で分かり、開発者の創作意欲を刺激する。

### ② アイデア詳細・投稿画面

- **構成:**
  - **Problem:** 何に困っているか？
  - **Solution:** どう解決したいか？（未定でも可）
  - **Status:** 「募集中」「開発中」「解決済み（URLあり）」のステータス表示。
- **UX:** 開発者が「Fork（このアイデアを自分なりに解釈して実装）」できる導線。

### ③ 実績プロフィール画面（The Museum of Achievement）

- **構成:**
  - **投稿者側:** 「生み出した価値」をグラフ化。採用されたアイデア数、それによる想定削減時間など。
  - **開発者側:** 解決したNeedのポートフォリオ。GitHubのリポジトリと連動。
- **UX:** 単なるスキルセットではなく「誰の、どんな課題を解決したか」というストーリーとしての履歴書。

## 4. 機能要件リスト

### MVP（実用最小限の機能）

| 機能 | 概要 |
|------|------|
| 認証基盤 | GitHub連携ログイン（開発者の信頼性担保） |
| Need投稿機能 | タイトル、内容、タグ付け |
| 「欲しい！」ボタン | 需要を数値化するためのリアクション機能 |
| ステータス管理 | 開発者が「着手宣言」を行い、進捗を表示する機能 |
| 成果物リンク | 完成したURLやリポジトリを紐付け、ステータスを「解決」にする |

### 将来的な追加機能

1. **称号（Badge）システム**
   - 「着眼点のプロ」: 投稿したNeedに100人以上が共感
   - 「爆速解決師」: 着手宣言から3日以内に解決

2. **インセンティブ設計**
   - 解決されたNeedに対して、投稿者が開発者に「サンクス・コーヒー（少額投げ銭）」を送れる仕組み

3. **AIサジェスト**
   - 投稿された文章から、適切な技術スタック（React, Pythonなど）を自動タグ付け

## 5. 技術スタック

ハッカソンでの「開発スピード」と「モダンな構成」を両立させます。

| レイヤー | 技術 | 理由 |
|----------|------|------|
| Frontend | Next.js (App Router) + Tailwind CSS + shadcn/ui | 高速な開発と、一貫性のある美しいUIを短時間で構築可能 |
| Backend/BaaS | Supabase | Auth・PostgreSQL・Storageが即座に揃い、リアルタイム機能の実装が容易 |
| Deployment | Vercel | Next.jsとの親和性が高く、プレビュー機能でチーム開発がスムーズ |
| Language | TypeScript | 型安全性により、ハッカソン特有の深夜のデバッグ作業を減らす |
