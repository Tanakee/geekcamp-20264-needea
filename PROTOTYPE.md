# NEEDEA 画面設計・機能仕様書

## サービスコンセプト

ニーディアは**エンジニアへの制作依頼サービスではない**。

「日々の不満」や「その人にしか気づけない面白いアイデア」をツイートするように気軽に投稿できる広場。技術知識は不要。「電車の遅延をもっと早く知りたい」「冷蔵庫の食材からレシピを提案してほしい」——そういった生活の中からにじみ出るリアルな声が集まることで、エンジニアが「作りたいもの」を見つける。

投稿者は発注者ではなく、**アイデアの発信者**。エンジニアは受注者ではなく、**共感した解決者**。この関係性がニーディアの核心。

---

## 共通：グローバルナビゲーション (Global Navigation Bar)

画面最上部に固定（Sticky）される、サービス全体の操作ハブです。

* **配置 (左端):** ロゴテキスト NEEDEA
* **配置 (中央):** ナビゲーションメニュー
  * Marketplace / Build-ers / Profile
* **配置 (右端):** 「ポストする」ボタン（アイデア投稿画面へ遷移）
* **カレント表示:** 現在のページの下に太いアンダーラインを引き、ユーザーが現在地を把握できるようにする

## 画面 1：Marketplace（トップ画面）

### 1-1. ヒーローセクション ＆ 認証 (Hero & Auth Section)

* **メインコピー:** 中央に大きく「あなたの不満は、誰かのコードになる。」を配置
* **認証ボタン:** GitHub でログイン ボタン（OAuth認証）
  * ログイン後はボタンが非表示になり、ユーザー専用コンテンツが表示される

### 1-2. 左カラム：Timeline / アイデアフィード（画面の約70%）

* **ヘッダー:** 🔥 Timeline タイトルの横にソート用ドロップダウンを配置
* **フィルタ機能:** 「Wants数（デフォルト：多い順）」「投稿順」の切り替え
* **スクロール挙動:** サイドバーは固定したまま、アイデアカードエリアのみ独立スクロール

#### アイデアカード（1行2列グリッド）

* **タイトル:** アイデア名を最上部に太字で配置
* **統計:** タイトルのすぐ下に「🔥 〇〇 Wants」（共感数）を表示
* **Wantボタン（🔥）:** カード右上に配置。押すとWants数がリアルタイムでカウントアップ（Optimistic UI）
* **詳細情報:** 左下に「@募集者名」＋ステータスバッジ（募集中 / 開発中 / 解決済み）
* **タグ:** ハッシュタグ形式で技術スタック・カテゴリを表示
* **Hover効果:** カード全体が浮き上がり（scale-up）、境界線が強調されるアニメーション

### 1-3. 右カラム：Community Sidebar（画面の約30%）

* **Featured Build-ers:** 貢献度の高いエンジニアのアバターと解決数を表示
* **Problem Solving Stats:** 解決済み・開発中・募集中の件数をバーグラフで可視化
* **Top Builders:** 解決数上位5名の円形アバターを一覧表示

### 1-4. テクニカルノート

* **Optimistic UI:** Wantボタン押下時、DB更新を待たずに即座にクライアント側の数字を増やす
* **State Management:** ナビゲーションのアクティブ状態はNext.jsのパス名で管理し、`border-bottom` で強調

## 画面 2：The Blueprint（アイデア詳細画面）

### 2-1. ページレイアウト

* **パンくず:** `The Blueprint / アイデア詳細`
* **2カラム構成:**
  * **左カラム（70%）:** アイデアの内容・不満の深掘り・コミュニケーション
  * **右カラム（30%）:** Build Panel（需要数値化・解決宣言・タグ管理）

### 2-2. 左カラム

#### ① 基本情報エリア

* **ステータスバッジ:** 募集中 / 開発中 / 解決済み
* **タイトル:** 大きく表示
* **投稿者情報:** アイコン＋@ユーザー名 を横並びで配置
* **困り度（Urgency Gauge）:** 1〜10の数値＋ゲージバー。10に近づくほど色が **青→黄→赤** に変化

#### ② 不満・解決策エリア

* **なぜ苦しているか（Problem / Needs）:** 枠線で囲まれたテキストエリア（必須）
* **解決策のProposal（Solution）:** 理想の解決方法。未定の場合は「開発者のアイデアを募集中！」と表示
* **解決済みの場合:** GitHubリポジトリ・成果物URLへのリンクを表示
* **具体例:** リスト形式。ユーザーケースを列挙（任意・最大5個）

#### ③ コメント欄

* **UI:** ユーザーアイコン＋メッセージ内容のリスト
* **機能:** 投稿者と開発者の意見交換の場。テキスト入力＋送信ボタン

### 2-3. 右カラム：Build Panel

#### ① Wants カウンター

* **UI:** 大きく「🔥 〇〇 Wants」を表示
* **「欲しい！」ボタン:** 押すとカウントアップ（Optimistic UI）

#### ② アクションパネル（Solve this!）

* **着手中エンジニア数:** リアルタイム表示
* **Solve this! ボタン:** クリックで着手人数が+1。クリック後は「着手済み ✓」に変化し、プロフィールの実績にカウント開始

#### ③ タグ管理

* **テキスト入力:** Enterでタグ追加
* **タグ表示:** 楕円形チップ形式で `#タグ名` を表示

### 2-4. テクニカルノート

* **Table: Ideas** — title, creator_id (uuid), urgency (1-10), core_problem (text), solution (text), examples (string[]), tags (string[])
* **Table: Solvers** — idea_id と user_id の紐付け（Solve this! を押したユーザーの記録）
* **動的カラーロジック:** `hsl(${240 - urgency * 24}, 100%, 50%)` で青から赤への変化

## 画面 3：アイデア投稿フォーム

### 3-1. 画面概要

* **タイトル:** 「日常の『あったらいいな』を世界へポストする」
* **2カラム構成:** 左にフォーム、右に「投稿のヒント・最近の投稿」サイドバー

### 3-2. 入力フォーム

| フィールド | 種別 | 仕様 |
|---|---|---|
| アイデア名（Title） | テキスト | 必須・最大50文字 |
| 困り度（Urgency） | ドロップダウン（1〜10） | 値に応じてUIカラーが変化（青/黄/赤） |
| 日常の不満（Problem） | テキストエリア | 必須 |
| 解決策のProposal（Solution） | テキストエリア | 任意・未定でも可 |
| 具体的な内容（Details） | 箇条書き入力 | 任意・最大10項目・動的追加 |
| タグ | チップ入力 | Enterまたはスペースでタグ化 |

### 3-3. ポストするボタン

* 必須項目未入力の場合は `disabled` 状態
* 送信後、トースト通知「✈️ 世界へポストされました！」を表示
* 投稿完了後、該当アイデアの詳細画面（The Blueprint）へリダイレクト

## 画面 4：メッセージ（DM機能）

投稿者と解決者が1対1でやり取りするためのコミュニケーション画面です。

* **宛先表示:** 「〇〇さんにメッセージを送る」
* **コンテキスト:** 対象のアイデアタイトル・Wants数を画面上部に表示
* **チャットUI:** アイコン＋メッセージのリスト形式
* **送信フォーム:** テキスト入力＋「メッセージを送信する」ボタン

---

## 画面 5：Build-ers 検索（アイデア解決者一覧）

エンジニアのスキルや実績を可視化し、最適な解決者を見つけるための画面です。

### 5-1. 検索・フィルタリング

* **技術スタック検索:** 使用言語・フレームワーク（React, Python など）による絞り込み
* **解決実績フィルタ:** 解決済みNeed数（Solved Needs）によるフィルタリング
* **バッジ・称号フィルタ:** 特定のバッジを保有するエンジニアの抽出
* **GitHub Verified トグル:** GitHub連携済みユーザーのみを表示

### 5-2. 表示コンテンツ

* **Featured Build-ers（注目解決者）:** 優秀なエンジニアをカード形式で紹介。解決数・平均日数・スキルタグを表示
* **おすすめの解決者（Solution Masters）:** マッチ度（%）に基づくランキングリスト
* **最近解決されたNeed:** 直近で解決された課題・担当エンジニア・GitHubリンク・称号を表示

---

## 画面 6：自分のプロフィール（アチーブメント・ミュージアム）

「課題発見者（Need-er）」と「課題解決者（Build-er）」双方の実績を証明するポートフォリオ画面です。

### 6-1. ユーザー統計

* **基本情報:** @ユーザー名・現在の称号・GitHubリンク
* **サマリー数値:** 投稿数・獲得Wants数・解決数・サンクス・コーヒー数

### 6-2. 実績グラフ

* **発想の価値グラフ（Need-er Stats）:** 投稿課題が集めたWants数・他者の時間削減量の集計グラフ
* **解決の実績グラフ（Build-er Stats）:** 解決課題数・総Wantスコア・カテゴリ別内訳の可視化

### 6-3. バッジ・ポートフォリオ

* **称号コレクション（The Badge Collection）:** 獲得済みバッジを表示。未獲得はグレーアウト
  * 🎯 着眼点のプロ（100人以上が共感）
  * ⚡ 爆速解決師（3日以内に解決）
  * 🌟 社会貢献の星（10件以上解決）
* **解決したNeedポートフォリオ（Story Resume）:** 過去に解決したプロジェクトの記録・解決ステータス・リポジトリURL
* **サンクス・コーヒー（少額投げ銭）:** 受け取った投げ銭履歴・感謝メッセージ・贈られた称号を表示

## 画面 7：フィードバック画面（Solution Presentation & Feedback）

提出された解決策を確認し、依頼者が評価・フィードバックを行ってプロジェクトを完結させる画面です。

### 7-1. 解決策の提示内容

* **ツール詳細:** 解決策の名称（例：EasyExcel）・概要説明・主要機能（Key features）のリスト
* **技術スタック:** 使用言語・ライブラリのハッシュタグ表示
* **成果物（Deliverables）:** GitHubリポジトリ・ライブデモ・スクリーンショット・動画デモへのアクセスボタン

### 7-2. フィードバックと承認アクション

* **承認確認:** 「解決を確認しました」に対して はい / いいえ で回答
* **コメント送信:** エンジニアへの感謝・フィードバックのテキスト入力
* **称号の授与:** 解決の質に応じてエンジニアにふさわしい称号を選択して贈る（例：初期化の魔術師）
* **リアクション:** 「すごい！」などのスタンプによる簡易評価（カウント表示）
